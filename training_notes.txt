Training Notes -- AudioGPT2 on EMoDB
=====================================

Dataset: EMoDB, 816 samples, 7 emotion classes
Split: speaker-independent (train: speakers 11-16 -> 493 samples, val: 09/10 -> 161, test: 03/08 -> 162)
Architecture: GPT-2 (frozen) + cross-attention fusion blocks that inject audio embeddings at intermediate layers


---

Run 1 -- First attempt
----------------------
Config: 30 epochs, lr=1e-4, 12 fusion blocks at every GPT-2 layer, adapter_dim=256, dropout=0.1
Trainable params: 33M

Results:
  Best checkpoint: epoch 4
  Test accuracy:   21.6%
  Weighted F1:     13.1%

Confusion matrix (rows=true, cols=pred):
  Labels: anger, boredom, disgust, fear, happiness, neutral, sadness

            ang  bor  dis  fea  hap  neu  sad
  anger   [  9    2    0    3    0    0   13 ]
  boredom [  0    0    0    0    0    0   20 ]
  disgust [  0    0    0    0    0    0   21 ]
  fear    [ 11    6    0    2    0    0    7 ]
  happy   [  5    5    0    0    0    0   12 ]
  neutral [  0    1    0    0    0    0   20 ]
  sadness [  0    1    0    0    0    0   24 ]

What happened:
The model basically gave up and predicted "sadness" for almost everything. Looking at the confusion
matrix, around 72% of all test samples got classified as sadness regardless of the true label.

The root cause was a massive overfitting problem. 33M trainable parameters on only 493 training
samples is way too many. Train loss collapsed to ~0.001 by epoch 20 while val loss blew up to 3.4+.
The best checkpoint was already at epoch 4, after which the model just memorized the training set.

On top of that, the learning rate (1e-4) was too high for this dataset size, which caused the loss
to crash fast rather than converge properly.


---

Run 2 -- Fixing overfitting
----------------------------
Changes from Run 1:
- Reduced fusion blocks from 12 to 4 (applied at GPT-2 layers 2, 5, 8, 11 instead of all 12)
- Reduced adapter_dim from 256 to 64
- Increased dropout from 0.1 to 0.3
- Lowered learning rate from 1e-4 to 1e-5
- Added class-weighted cross entropy loss (computed from training split labels)
Trainable params: 10M

Results:
  Best checkpoint: epoch 30
  Test accuracy:   35.8%
  Weighted F1:     28.7%

Train/val loss at best checkpoint (epoch 30): train=1.8834, val=1.8860 -- almost no gap, good.

Confusion matrix (rows=true, cols=pred):
  Labels: anger, boredom, disgust, fear, happiness, neutral, sadness

            ang  bor  dis  fea  hap  neu  sad
  anger   [  9    0    0    9    0    7    2 ]
  boredom [  0    0    1    0    0   12    7 ]
  disgust [  0    0    1    1    0   11    8 ]
  fear    [  1    0    0   20    0    4    1 ]
  happy   [  2    0    1   13    0    4    2 ]
  neutral [  0    0    0    4    0   15    2 ]
  sadness [  0    0    0    2    0   10   13 ]

What happened:
Big improvement. The class collapse was gone and all 7 emotions were appearing in the predictions.
The overfitting was fixed too -- train and val loss stayed very close to each other the entire run
(both around 1.88-1.89).

The issue now was that the best checkpoint was the very last epoch, meaning the model was still
actively learning when we stopped. 30 epochs just wasn't enough.


---

Run 3 -- More training time
-----------------------------
Changes from Run 2:
- Increased epochs from 30 to 150 (same architecture and hyperparameters otherwise)

Results:
  Best checkpoint: epoch 81
  Test accuracy:   49.4%
  Weighted F1:     46.8%

Train/val loss at best checkpoint (epoch 81): train=0.6642, val=1.4204 -- gap is opening up,
overfitting starts here. After this the val loss goes back up while train keeps falling.

Confusion matrix (rows=true, cols=pred):
  Labels: anger, boredom, disgust, fear, happiness, neutral, sadness

            ang  bor  dis  fea  hap  neu  sad
  anger   [ 18    0    3    2    2    0    2 ]
  boredom [  0    7    5    0    0    3    5 ]
  disgust [  1    4    9    0    0    0    7 ]
  fear    [  9    0    1   11    2    0    3 ]
  happy   [ 12    0    1    5    2    2    0 ]
  neutral [  0    4    1    2    2   12    0 ]
  sadness [  0    0    1    0    0    3   21 ]

Per-class accuracy at best checkpoint:
  Sadness:   21/25 = 84%
  Anger:     18/27 = 67%
  Neutral:   12/21 = 57%
  Disgust:    9/21 = 43%
  Fear:      11/26 = 42%
  Boredom:    7/20 = 35%
  Happiness:  2/22 = 9%

What happened:
Another big jump. The confusion matrix looks much healthier now with no dominant predicted class.
Sadness and anger are learned well, neutral is decent, and the others are still mediocre but at
least being predicted.

The main problem left is happiness. 12 out of 22 happiness samples got classified as anger.
This is a known issue with EMoDB -- both are high-arousal emotions with similar pitch and energy
in German speech, so they are acoustically hard to separate.

Also worth noting: overfitting started around epoch 81. After that, train loss kept falling
(0.66 -> 0.40) while val loss went up (1.42 -> 1.72). So the last 69 epochs were basically wasted.
Next run should stop around 100 epochs.


---

Run 4 -- Swapping encoder to WavLM-large
------------------------------------------
Changes from Run 3:
- Replaced encoder from facebook/wav2vec2-base-960h (768-dim) to microsoft/wavlm-large (1024-dim)
- Added audio_dim parameter to AudioGPT2 (was hardcoded to 768) so the fusion blocks instantiate
  audio_projection as Linear(1024, 768) instead of Identity -- four blocks x 787K params each
- epochs reduced from 150 to 100 (run 3 showed overfitting started around epoch 81)
- All other hyperparameters unchanged (lr=1e-5, adapter_dim=64, dropout=0.3)
Trainable params: 13.2M (up from 10M, difference is the four audio_projection layers)

Results:
  Best checkpoint: epoch 33
  Test accuracy:   88.3%
  Weighted F1:     88.2%

Train/val loss at best checkpoint (epoch 33): train=0.2852, val=0.6180 -- gap already opening.
By epoch 100: train=0.004, val=0.883 -- severe overfitting in the second half.

Confusion matrix (rows=true, cols=pred):
  Labels: anger, boredom, disgust, fear, happiness, neutral, sadness

            ang  bor  dis  fea  hap  neu  sad
  anger   [ 27    0    0    0    0    0    0 ]
  boredom [  0   15    2    0    0    0    3 ]
  disgust [  0    1   18    1    1    0    0 ]
  fear    [  0    0    0   23    3    0    0 ]
  happy   [  3    0    0    1   18    0    0 ]
  neutral [  0    1    0    0    0   20    0 ]
  sadness [  0    3    0    0    0    0   22 ]

Per-class accuracy at best checkpoint:
  Anger:     27/27 = 100%
  Neutral:   20/21 = 95%
  Disgust:   18/21 = 86%
  Sadness:   22/25 = 88%
  Fear:      23/26 = 88%
  Happiness: 18/22 = 82%
  Boredom:   15/20 = 75%

What happened:
Massive jump. The single change was the encoder -- everything else was identical to run 3.
This confirmed that wav2vec2-base was the bottleneck the whole time, not the architecture or
the hyperparameters. WavLM-large was pretrained on 94k hours of diverse speech with both masked
prediction and denoising objectives, which incidentally captures prosodic and paralinguistic
features that emotion recognition depends on.

The happiness vs anger confusion that plagued runs 1-3 is almost gone. Run 3 had 12 happiness
samples misclassified as anger; here it's down to 3. WavLM's richer feature space separates
the two high-arousal emotions that wav2vec2 couldn't distinguish.

Overfitting pattern shifted earlier compared to run 3. Best checkpoint was epoch 33 instead of
epoch 81, and train loss hit near-zero (0.004) by epoch 100 while val loss climbed from 0.62
to 0.88. The model has 13.2M trainable parameters on 493 training samples, which is an even
worse ratio than before because of the added projection layers. The early stopping via best
checkpoint saved the result; without it the final model would have been much worse.

Run 5 -- wav2vec2-large-emotion encoder
-----------------------------------------
Changes from Run 4:
- Replaced encoder from microsoft/wavlm-large to audeering/wav2vec2-large-robust-12-ft-emotion-msp-dim
- Both are 1024-dim so no architecture changes needed
- All other hyperparameters unchanged (lr=1e-5, adapter_dim=64, dropout=0.3, 100 epochs)
Trainable params: 13.2M

Results:
  Best checkpoint: epoch 38 (approx)
  Test accuracy:   93.2%
  Weighted F1:     93.2%

What happened:
Best result across all runs. The encoder was fine-tuned on MSP-Podcast emotional speech labels,
so its representations are already organized around emotion -- the downstream model has less work
to do. This confirms that encoder pre-training objective matters more than encoder size: a
1024-dim emotion-tuned model outperforms a 1024-dim general-purpose model (WavLM-large, 88.3%).

AudioGPT2 also outperforms the MLP probe on this encoder (MLP probe: 91.4%), which is the only
case across all encoders where the architecture adds measurable value over a simple classifier.


---

Run 6 -- HuBERT-large encoder
-------------------------------
Changes from Run 4:
- Replaced encoder from microsoft/wavlm-large to facebook/hubert-large-ls960-ft (1024-dim)
- All other hyperparameters unchanged
Trainable params: 13.2M

Results:
  Test accuracy:   78.4%
  Weighted F1:     78.6%

What happened:
HuBERT falls between wavlm-large (88.3%) and wav2vec2-base (42.6%). Both HuBERT and WavLM use
masked prediction pretraining on clean speech, but WavLM adds a denoising objective which may
incidentally capture prosodic variation useful for emotion. The ranking among general-purpose
encoders is: WavLM-large > HuBERT-large >> wav2vec2-base.

LoRA experiments -- WavLM-large encoder
-----------------------------------------
Motivation: GPT-2 layers between fusion blocks apply a frozen, task-irrelevant transformation.
LoRA (low-rank adaptation) allows GPT-2's attention projections to adapt to the task without
full fine-tuning. Applied rank-8 LoRA to c_attn and c_proj in all 12 GPT-2 layers (~442K
extra trainable parameters). All LoRA runs use wavlm-large as encoder.

LoRA Run 1 -- uniform learning rate
  Config: lora_rank=8, all parameters at lr=1e-5 (no separate LoRA LR)
  Best checkpoint: epoch 38
  Test accuracy:   86.4%
  Weighted F1:     86.3%
  Result: worse than frozen baseline (88.3%). LoRA parameters barely moved at 1e-5 -- too
  small to adapt meaningfully, but enough to slightly disturb the adapter's learned alignment.

LoRA Run 2 -- higher LoRA learning rate
  Config: lora_rank=8, adapter lr=1e-5, lora_lr=1e-4
  Result: unstable. Val loss oscillated wildly after epoch 22. Train loss collapsed to ~0.009
  by epoch 38 while val loss bounced between 0.67 and 0.87. The 10x LR difference caused LoRA
  to shift GPT-2's representations faster than the adapters could follow. No test results.

LoRA Run 3 -- intermediate LoRA learning rate
  Config: lora_rank=8, adapter lr=1e-5, lora_lr=2e-5
  Best checkpoint: epoch 26 (val_loss: 0.7246, val_acc: 74.5%)
  Test accuracy:   [fill in]
  Weighted F1:     [fill in]
  Training was stable but overfitting still present -- train loss near-zero by epoch 35 while
  val loss oscillated without improving past epoch 26.

Conclusion:
LoRA did not improve over the frozen baseline on any of the three learning rate settings.
493 training samples is too few for the backbone to adapt without overfitting, even with a
low-rank constraint. The dataset size is the fundamental bottleneck; LoRA is the right
architectural idea but requires a larger corpus to be effective.

Baselines
---------
All three use the same speaker-independent split (test: speakers 03, 08, 162 samples).


SVM + MFCC
-----------
Config: 84 hand-crafted features (40 MFCCs mean+std, F0 mean+std of voiced frames, RMS energy
mean+std). SVC with kernel=rbf, C=10.0, gamma=scale, class_weight=balanced.

Results:
  Test accuracy:   63.6%
  Weighted F1:     63.4%

Confusion matrix:
  Labels: anger, boredom, disgust, fear, happiness, neutral, sadness

            ang  bor  dis  fea  hap  neu  sad
  anger   [ 24    0    0    1    2    0    0 ]
  boredom [  0   11    3    0    0    4    2 ]
  disgust [  0    1   12    6    2    0    0 ]
  fear    [  7    0    2   13    4    0    0 ]
  happy   [  5    0    3    1   13    0    0 ]
  neutral [  1    1   11    0    0    8    0 ]
  sadness [  0    2    1    0    0    0   22 ]

What happened:
Strong anger (89% recall) and sadness (88% recall) but neutral is badly confused with disgust
(11 out of 21 neutral samples). The hand-crafted features don't separate low-arousal emotions
cleanly. Val accuracy was 50.3%, so the model generalized to test but there was not much room to go.

Linear and MLP probes
----------------------
A linear classifier and a small MLP trained on top of frozen encoder embeddings. No fine-tuning of
the encoder. Run for all four encoders to measure how much emotion structure the representations
contain before any task-specific adaptation.

Results (accuracy / weighted F1):

                            linear probe          MLP probe
  wav2vec2-base               49.4% / 47.3%       42.0% / 40.5%
  hubert-large                75.3% / 75.7%       81.5% / 81.4%
  wav2vec2-large-emotion      85.2% / 85.1%       85.8% / 86.1%
  wavlm-large                 87.0% / 87.1%       92.6% / 92.5%

What happened:
wav2vec2-base is the outlier in both directions -- the MLP probe (42.0%) is actually worse than
the linear probe (49.4%), which typically means the embedding space has no separable emotion
structure that a nonlinear model can exploit without overfitting. wavlm-large and
wav2vec2-large-emotion are completely different: both sit above 85% with just a linear layer.

The wavlm-large MLP probe reaching 92.6% while AudioGPT2 with the same encoder reached 88.3%
confirms that the cross-attention architecture is not adding value on this dataset size. A model
with far fewer trainable parameters wins. The wav2vec2-large-emotion MLP probe (85.8%) is the
exception where AudioGPT2 clearly wins (93.2%), likely because the encoder's representations are
already organized around emotion labels, giving the adapters a cleaner signal to work with.

Full comparison (test set)
---------------------------

                        hubert-large   wav2vec2-base   wav2vec2-large-emotion   wavlm-large
  SVM + MFCC                  63.6%           63.6%                    63.6%         63.6%
  linear probe                75.3%           49.4%                    85.2%         87.0%
  MLP probe                   81.5%           42.0%                    85.8%         92.6%
  AudioGPT2                   78.4%           42.6%                    93.2%         88.3%

AudioGPT2 beats the MLP probe only on wav2vec2-large-emotion. On every other encoder the probe
wins or ties. The pattern is consistent: the architecture needs more data to justify its parameter
count. On EMoDB the encoder is the only lever that reliably moves the number.
